# Jux Spec Addendum — Bindings and Interface Stubs (`juxc bindgen`)

**Status:** Proposed insertion. Specifies how foreign APIs (Rust crates, Rust `std`, C headers, C++ headers) are surfaced to Jux source and to editor tooling as **Jux-syntax interface stubs** (`.jux.d` files), the type-mapping rules `juxc bindgen` applies, the naming transforms, the resolver ingestion mode that consumes stubs, and the contract that makes hover/completion/go-to-definition work for foreign symbols without a separate indexer. Companion to `JUX-LSP-SERVER-ADDENDUM.md` (which consumes the resolver's output) and `JUX-COMPILER-PIPELINE-ADDENDUM.md` (which owns name resolution, phase 4).

## Design Philosophy (Non-Normative)

Jux does not ship a large standard library. Per `JUX-CORE-LIB-ADDENDUM.md` §K.1, only the types the compiler synthesizes calls to, relies on the layout of, or whose syntax depends on them, live in `core` — `Option<T>`, `Result<T, E>`, `Iterable<T>`, `String`, `StackString<N>`, and a short list besides. Everything richer — collections, hashing, formatting, networking, JSON, async runtimes — is **Rust**, reached through the FFI layers of JUX-LANG-V1 §8.2.

This is the load-bearing decision: **Rust is the standard library; Jux is the syntax layer on top of it.** A Jux program that wants a hash map uses Rust's `HashMap`; one that wants JSON uses `serde_json`; one that wants HTTP uses `reqwest`. The Jux compiler already lowers to Rust (Phase 1, §2.2), so these crates are not foreign at the binary level — they are foreign only at the *source* level.

What is missing, and what this addendum supplies, is the bridge that lets a programmer **write** and **read** those Rust APIs in Jux syntax — with IDE autocomplete, hover types, and go-to-definition — without learning Rust's spelling. That bridge is a set of **interface stubs**: signature-only Jux declarations, one per foreign type or function, generated by `juxc bindgen` and consumed by the ordinary name resolver.

Three consequences shape the rest of this addendum:

1. **Stubs are Jux, not a side-format.** A `.jux.d` file is a valid Jux declaration file whose bodies are elided (`;` per the grammar's `function-body` production, `JUX-GRAMMAR-ADDENDUM.md` §A.2.4). The resolver loads it exactly as it loads any module. There is no second parser, no TypeScript-style separate `.d.ts` grammar, no bespoke LSP indexer.
2. **Editor support falls out for free.** Because stubs resolve through the same `juxc-resolve` and `juxc-tycheck` crates that drive batch compilation, `juxc-lsp` (per `JUX-LSP-SERVER-ADDENDUM.md`) answers hover, completion, and goto-def for `HashMap` the same way it answers them for a user's own class. No LSP code is specific to bindings.
3. **The mapping is mechanical and pinned.** Every translation rule in this addendum is derived from a rule already stated elsewhere in the spec — the type table of §8.2, the construction-operator rule of §7.3.1, the naming conventions of §3.3, the hybrid-paradigm guarantee of §1.1.6. `juxc bindgen` invents nothing; it applies the spec's existing translations in bulk.

## §G.1 — Goals and Non-Goals

### Goals

| # | Capability                                                              |
|---|-------------------------------------------------------------------------|
| 1 | Generate Jux-syntax `.jux.d` stubs from a Rust crate's public API       |
| 2 | Generate `.jux.d` stubs (plus a Rust FFI shim) from a C header          |
| 3 | Generate `.jux.d` stubs (plus an `autocxx`-backed shim) from a C++ header |
| 4 | Apply the §8.2 type-mapping table uniformly and reversibly              |
| 5 | Apply snake_case → camelCase / PascalCase naming transforms (§3.3)      |
| 6 | Emit stubs the ordinary resolver can ingest with no special parser      |
| 7 | Enable hover / completion / goto-def on foreign symbols via `juxc-lsp`  |

### Non-Goals

- **Generating bodies.** Stubs are signatures only. The implementation lives in the Rust crate / C library; the Jux side never sees a body.
- **Importing Rust macros.** Per §8.2, macros are not importable. `bindgen` skips them.
- **Round-tripping Jux → Rust source.** This addendum covers the foreign → Jux direction. Exporting Jux to C is `@export` (§8.4), a separate mechanism.
- **A second type system.** Stubs are checked by the same `juxc-tycheck` as everything else. There is no "binding type system."
- **Runtime reflection over foreign types.** Out of scope; see §G.13.

## §G.2 — The Stub File (`.jux.d`)

### G.2.1. What a Stub File Is

A `.jux.d` file is a Jux source file restricted to **declarations with elided bodies**. It contains package and import declarations, type declarations (`class`, `interface`, `struct`, `record`, `enum`), free-function signatures, and constants — but every function body is a single `;`, and every field is a declaration without an initializer.

The `.jux.d` extension distinguishes stubs from ordinary `.jux` sources so the build system can treat them as **external/opaque** (§G.9): they contribute signatures to name resolution and type checking but never reach MIR construction, borrow inference, or codegen.

### G.2.2. Grammatical Basis

No grammar extension is required. Stub files use the existing productions:

- Elided function bodies use `function-body = ';'` (`JUX-GRAMMAR-ADDENDUM.md` §A.2.4), already legal for `abstract` and `native` declarations.
- Elided constructors use the same `;` body.
- Fields without initializers use `field-decl` with the `( '=' expression )?` clause omitted.

A stub file is therefore parseable by the unmodified `juxc-parse`. The only new behavior is at the resolver level (§G.9), gated by the `.jux.d` extension.

### G.2.3. Worked Example

```jux
// stubs/std.collections.jux.d   — generated by `juxc bindgen --rust std`
package rust.std.collections;

public class HashMap<K, V> {
    public HashMap();                       // Rust HashMap::new()
    public HashMap(int capacity);           // Rust HashMap::with_capacity(usize)

    public void insert(K key, V value);
    public V? get(K key);                   // Rust get(&K) -> Option<&V>
    public bool containsKey(K key);
    public void remove(K key);
    public int len();
    public bool isEmpty();
    public Iterable<K> keys();
    public Iterable<V> values();
}
```

Every body is `;`. The file is valid Jux. The resolver loads it; `juxc-tycheck` records the type of every signature; `juxc-lsp` answers questions about `HashMap` from that recorded type.

## §G.3 — Type Mapping

`juxc bindgen` maps foreign types to Jux types using the table from JUX-LANG-V1 §8.2, restated here as the normative `bindgen` contract and extended with the cases §8.2 left implicit.

### G.3.1. Rust → Jux

| Rust type                     | Jux type                           | Notes                                              |
|-------------------------------|------------------------------------|----------------------------------------------------|
| `i8` / `i16` / `i32` / `i64`  | `byte` / `short` / `i32` / `long`  | width-explicit forms preserved where load-bearing (§5.1) |
| `u8` / `u16` / `u32` / `u64`  | `ubyte` / `ushort` / `u32` / `ulong` |                                                    |
| `isize` / `usize`             | `int` / `uint`                     | platform-sized maps to platform-sized (§5.1)       |
| `f32` / `f64`                 | `float` / `double`                 |                                                    |
| `bool`                        | `bool`                             |                                                    |
| `char`                        | `char`                             | Unicode scalar, 32-bit (§5.1)                      |
| `String`, `&str`              | `String`                           | §8.2                                               |
| `Vec<T>`                      | `List<T>`                          | §8.2                                               |
| `HashMap<K, V>`, `BTreeMap<K,V>` | `Map<K, V>`                     | §8.2 (both map to the one canonical `Map`)         |
| `HashSet<T>`, `BTreeSet<T>`   | `Set<T>`                           | §8.2                                               |
| `Option<T>`                   | `T?`                               | §8.2; nullable type                                |
| `Result<T, E>`                | return type `T throws E`           | §8.2; see §G.5.4                                   |
| `Box<T>`, `Rc<T>`, `Arc<T>`   | `T`                                | refcount managed by Jux (§8.2)                     |
| `&T`, `&mut T`                | `T`                                | borrow inferred at use site (§8.2, §6.3)           |
| `Box<dyn Trait>`              | `Trait`                            | Jux interface (§8.2)                               |
| `impl Trait` (return)         | `Trait`                            | the named trait's Jux interface                    |
| `Fn(A) -> B` family           | `(A) -> B`                         | function type (§8.2, §7.9)                         |
| `(A, B)`                      | `Tuple<A, B>`                      | tuple (§5.3); nominal placeholder, see §G.3.5      |
| `[T; N]`                      | `T[N]`                             | const-size array (§5.5)                            |
| `&[T]`                        | `T[]`                              | runtime-size array                                 |
| `()` (unit)                   | `void`                             | in return position                                 |
| `!` (never)                   | `never`                            | bottom type (§K.4.1)                               |
| `*const T`, `*mut T`          | `Ptr<T>`                           | raw pointer (§5.5); nominal placeholder, see §G.3.5 |
| Macros                        | — (skipped)                        | not importable (§8.2)                              |

### G.3.2. Nullability and `Option`

A Rust `Option<T>` in a **return position** maps to `T?` (the canonical Jux nullable, §7.10). A Rust `Option<T>` appearing **nested** inside another generic (e.g. `Vec<Option<T>>`) maps to `List<T?>`. The two Jux spellings `T?` and `Option<T>` are distinct types (§K.3.1); `bindgen` always emits `T?` for clarity and never the `Option<T>` spelling.

### G.3.3. Errors and `Result`

A Rust function returning `Result<T, E>` maps to a Jux function with return type `T` and a `throws E` clause (§8.2, §7.11). The error type `E` is mapped through this same table; if `E` is a Rust enum, it becomes a Jux `enum` stub (§G.6.3). See §G.5.4 for the signature shape.

### G.3.4. Borrows Vanish

Rust's `&T` and `&mut T` carry no Jux spelling — they map to plain `T`, and Jux's borrow inference (§6.3, `JUX-TYPE-SYSTEM-ADDENDUM.md` §T.7) re-derives the borrow mode at each call site. This is the central reason Jux code reads like Java even though it lowers to borrow-checked Rust: the `&` is inferred, never written. A `&mut self` method becomes an ordinary mutating method (§6.3); a `&self` method becomes a non-mutating one.

### G.3.5. Phase-1 Nominal Placeholders — Tuples and Raw Pointers

The grammar reserves `tuple-type` (`(A, B)`, §A.2.7) and `pointer-type` (`T*`, §A.2.7, `unsafe`-only) but the Phase-1 parser does not yet read either spelling back in. Because a generated stub must **parse** for its enclosing member to survive into the symbol table (and thus autocomplete), `bindgen` surfaces these two types under nominal placeholders until the real type-syntax features land (the pointer/tuple work travels with the broader `unsafe` / C-interop effort):

- a tuple `(A, B, …)` → `Tuple<A, B, …>` (the unit tuple `()` still folds to `void` in return position);
- a raw pointer `*const T` / `*mut T` → `Ptr<T>` (a pointer to unit, `*mut ()` — the common opaque `void*` — has a `Void` pointee, which is not a valid type argument, so it surfaces as `Ptr<Object>` rather than the unparseable `Ptr<void>`).

Both spellings keep the element/pointee types visible in signatures, hover, and completion. `Tuple<…>` / `Ptr<T>` are not declared types in Phase 1 — they read as opaque foreign names, which is harmless because a stub's own signatures are never validated (§G.9.1). When `tuple-type` / `pointer-type` gain parser and lowering support, this section is removed and the §G.3.1 rows revert to `(A, B)` / `T*`.

## §G.4 — Naming Transforms

`juxc bindgen` rewrites foreign names to Jux conventions (§3.3). The transforms are deterministic and reversible (the shim records the original symbol).

| Foreign convention            | Jux convention      | Transform                                  |
|-------------------------------|---------------------|--------------------------------------------|
| Rust `snake_case` fn/method   | `camelCase`         | `with_capacity` → `withCapacity`           |
| Rust `PascalCase` type        | `PascalCase`        | unchanged (`HashMap` → `HashMap`)          |
| Rust `SCREAMING_SNAKE` const  | `SCREAMING_SNAKE`   | unchanged                                  |
| Rust module path `a::b::c`    | package `a.b.c`     | `::` → `.` (§4.2)                          |
| C `Lib_function` prefix       | `lib.function`      | prefix stripped, namespaced under class (§G.7) |
| C `STRUCT_FIELD`              | `field`             | lower-cased, de-prefixed                   |

### G.4.1. Collisions

If two foreign symbols transform to the same Jux name (e.g. Rust `to_string` and `toString` both → `toString`), `bindgen` keeps the first in declaration order and emits the second with a numeric suffix plus a `W`-class warning (§G.12). The user may pin an explicit Jux name in the `bindgen` config (§G.11).

### G.4.2. Reserved-Word Escaping

A foreign name that transforms to a Jux reserved keyword (§3.2) is suffixed with an underscore: a Rust method `type()` becomes Jux `type_()`. The shim maps it back.

## §G.5 — Member Mapping

This is the section the hybrid-paradigm guarantee (§1.1.6) governs: Jux is **OOP and procedural, freely mixed** (§7.17), so `bindgen` emits both instance members and free functions, choosing the shape that matches the foreign declaration.

### G.5.1. Constructors — `new`

Rust's conventional constructor `T::new()` maps to a Jux constructor, invoked with the `new` operator (§7.3.1):

```jux
// Rust: HashMap::new()
var m = new HashMap<String, int>();
```

The stub declares it as a constructor (method name = class name per §7.3.1, body `;`):

```jux
public class HashMap<K, V> {
    public HashMap();              // ← Rust ::new()
}
```

### G.5.2. Named Constructors — Other Associated Functions Returning `Self`

A Rust associated function (no `self` receiver) that returns `Self` and is *not* named `new` maps to an **overloaded constructor** when its parameter list is distinct, or to a **static method** when overloading would be ambiguous:

```jux
// Rust: HashMap::with_capacity(n)  → overloaded constructor
var m = new HashMap<String, int>(256);

// Rust: Duration::from_secs(s)     → static method (multiple from_* would collide as constructors)
var d = Duration.fromSecs(5);
```

The decision rule: if the type has exactly one non-`new` `Self`-returning associated function, or several with non-overlapping parameter lists, emit constructors. If several share a parameter shape (the common `from_secs` / `from_millis` / `from_nanos` case), emit static methods to preserve the distinct names. `bindgen` chooses; the config can override per-symbol (§G.11).

### G.5.3. Instance Methods

A Rust method with a `self` / `&self` / `&mut self` receiver maps to a Jux instance method. The receiver disposition is dropped (it becomes borrow-inferred, §G.3.4):

```jux
// Rust: fn insert(&mut self, k: K, v: V)
public void insert(K key, V value);     // mutating — inferred from &mut self
// Rust: fn get(&self, k: &K) -> Option<&V>
public V? get(K key);                    // non-mutating — inferred from &self
```

### G.5.4. `Result`-Returning Methods

A method returning `Result<T, E>` gets a Jux `throws E` clause (§G.3.3):

```jux
// Rust: fn parse(s: &str) -> Result<Config, ConfigError>
public Config parse(String s) throws ConfigError;
```

In exception-disabled profiles the compiler lowers `throws E` back to `Result<T, E>` (§7.11, §16.7) — so the stub is profile-agnostic, matching the source-portability promise.

### G.5.5. Free Functions

A Rust free function (module-level, no associated type) maps to a Jux **free function** (§7.17), preserving the procedural style:

```jux
// Rust: serde_json::from_str::<Value>(s) -> Result<Value, Error>
public Value fromStr(String s) throws JsonError;
```

The user calls it as a free function or imports it by name (§4.2):

```jux
import rust.serde_json.{Value, fromStr};
var v = fromStr(input);            // procedural style, no object
```

### G.5.6. Static Constants

A Rust `const` or `static` maps to a Jux `public const` (§A.2.2), typed through §G.3:

```jux
// Rust: std::f64::consts::PI
public const double PI = 3.141592653589793;
```

### G.5.7. Generic Bounds

A Rust trait bound `where T: Ord` maps to the Jux structural-constraint form (§7.8): `where T has operator<=>(T) -> int`. Common Rust traits map to their operator-or-interface equivalents:

| Rust bound      | Jux constraint                          |
|-----------------|-----------------------------------------|
| `T: Clone`      | (none — value copy is implicit; classes give a copy method, §K.2.2) |
| `T: PartialEq`  | `T has operator==(T) -> bool`           |
| `T: Ord`        | `T has operator<=>(T) -> int`           |
| `T: Hash`       | `T has operator hash() -> int`          |
| `T: Display`    | `T has operator string() -> String`     |
| `T: Iterator`   | `T : Iterable<E>`                       |
| `T: SomeUserTrait` | `T : SomeUserTrait` (interface, §G.6.4) |

## §G.6 — Rust Crate Bindings

### G.6.1. Source of Truth

`juxc bindgen --rust <crate>` reads the crate's public API from `rustdoc`'s JSON output (`rustdoc --output-format json`), not from source parsing. This gives `bindgen` fully-resolved types, trait bounds, and visibility without reimplementing a Rust front end.

### G.6.2. Pipeline

```
crate source
    ↓  cargo + rustdoc --output-format json
rustdoc JSON  (fully-resolved public API)
    ↓  juxc bindgen --rust
<crate>.jux.d   (Jux-syntax stubs, §G.2)
    +
no shim needed — Jux already lowers to Rust (§2.2), so the
call lowers directly to the crate's real symbol.
```

Because Phase 1 lowering already emits Rust (§C.9), a bound Rust crate needs **no FFI shim** — the generated Rust simply names the crate's items directly. The stub is purely an IDE/resolver artifact; it carries no runtime cost.

### G.6.2.1. The Default `rust.std` Surface — Generated, Not Curated

The Rust standard library is auto-loaded into every compile and editor analysis (the §G.3 mirror of the implicit `jux.std/` prepend), so `Vec` / `HashMap` / `String` autocomplete in Jux syntax with no `requires`. Crucially, **no std `.jux.d` is checked into the repository** — `juxc` generates the `rust.std` stub from the **toolchain the user actually has installed**, so the surface always matches their compiler:

- **Source.** The pre-built rustdoc JSON the `rust-docs-json` rustup component ships under `<sysroot>/share/doc/rust/json/` (located via `rustc +nightly --print sysroot`, then the default toolchain). When that component is absent, std autocomplete is simply unavailable — never an error.
- **Merge set.** `alloc` + `std` are ingested and merged into one `rust.std` package via the cross-crate, name-deduplicated path (§G.6.2.2). `core` is **excluded**: its JSON is ~20k items / tens of MB (operator traits, SIMD, every primitive's intrinsics), far too heavy to lex on every compile, and the only `core` types a Jux programmer reaches — `Option`, `Result` — are folded away by the type map (§G.3.1). A user who needs the rest of `core` binds it on demand as an ordinary `rust.core` dependency (§G.11).
- **Cache.** The generated stub is written once to the OS user-cache dir (`<cache>/juxc/stubs/rust-std.jux.d`) and loaded directly thereafter — no subprocess on the hot path, so the LSP's per-keystroke `check_workspace` stays cheap. A version marker in the cache header invalidates it when the bindgen surface changes; deleting the cache forces regeneration after a toolchain update.
- **Override.** `$JUX_STUBS_DIR` points the loader at a directory of `.jux.d` files loaded verbatim (no generation) — the hook for tests and for vendoring a frozen std surface.

### G.6.2.2. Merging Layered Crates

A single crate's rustdoc JSON only fully defines its **own** (`crate_id == 0`) items; items it merely re-exports from a lower layer appear as external references and are skipped. Rust's std is layered `core` ⊂ `alloc` ⊂ `std` — `Vec`, `String`, `Box`, `Rc`/`Arc`, `BTreeMap` are *defined* in `alloc` and only re-exported by `std` — so ingesting `std` alone misses them. `bindgen` therefore ingests each crate's JSON in turn (each as the local crate) and merges the results into one package, keyed by item name with **first-definition-wins** (crates supplied most-fundamental-first). Deduplication also collapses the platform-duplicated names std ships (e.g. the several `ChildExt` traits under `std::os::*::process`) that would otherwise collide as duplicate Jux declarations (`E0400`).

### G.6.3. Type Kind Selection

| Rust item        | Jux stub kind                                      |
|------------------|----------------------------------------------------|
| `struct` (public fields) | `struct` (§5.2) if all fields public & Copy-like; else `class` |
| `struct` (private fields, methods) | `class`                                  |
| `enum` (C-like)  | `enum` with explicit discriminators (§7.7.1 Pattern B) |
| `enum` (payloads)| `enum` with payload variants (§7.7.1 Pattern C)    |
| `trait`          | `interface` (§5.2)                                 |
| `type` alias     | `type` alias (§5.7)                                |
| `fn`             | free function (§G.5.5)                              |

### G.6.4. Traits → Interfaces

A Rust `trait` becomes a Jux `interface`; its required methods become abstract methods, its provided methods become `default` methods (§7.4.3):

```jux
// Rust: trait Read { fn read(&mut self, buf: &mut [u8]) -> Result<usize>;
//                    fn read_to_end(&mut self, ...) -> Result<usize> { default } }
public interface Read {
    int read(byte[] buf) throws IoError;
    default int readToEnd(List<byte> buf) throws IoError;
}
```

#### G.6.4.1. Signature-Only Surfacing of Bodied / Static / Valued Members

A `.jux.d` stub is **signature-only** (§G.2), which constrains how three Rust constructs surface — each would otherwise require a body Jux's grammar mandates but a stub never has:

- **Provided trait methods** (a Rust default impl) are surfaced as **plain abstract signatures**, not Jux `default` methods. A Jux `default` method requires a body (`E0200`); the provided-ness isn't representable bodyless and is moot for a never-lowered stub. (The example above shows the *intent*; the emitted `.jux.d` drops the `default` keyword.)
- **Associated functions without a receiver** on a trait (Rust "static" trait methods) are surfaced as **plain abstract signatures**, not Jux `static` interface methods — a bodyless `static` interface method is likewise `E0200`. (On a *class* stub, `static` is kept: a class may carry a bodyless static signature.)
- **Constants / statics** are surfaced as **bodyless** `const T NAME;` declarations. The rustdoc initializer is a *Rust* expression (`crate::sys::path::SEP`, `'\\'`, a const-fn call) with no Jux spelling; since a stub const is never lowered (§G.9) the value carries no information, so it is elided. The parser accepts an initializer-less `const` exactly for this stub form (§A.2.2 / §G.2).

These are surfacing choices, not language changes: the real crate provides every body, value, and static dispatch at link time.

### G.6.5. First-Class `import rust.X`

Per §8.2 Layer 3, the long-term path is the compiler reading Rust signatures directly. `bindgen`-generated `.jux.d` files are the Phase-1/Phase-2 realization of that: `import rust.serde_json.Value` resolves to the `Value` declaration in the generated `serde_json.jux.d`. When Layer 3 lands, the same import surface is served by an in-compiler reader instead of a pre-generated file; **the Jux-facing spelling does not change.**

## §G.7 — C Header Bindings

### G.7.1. Pipeline

```
mylib.h
    ↓  libclang (parse the header)
    ↓  juxc bindgen --header
mylib.jux.d        (Jux-syntax stubs)
    +
mylib_sys.rs       (Rust extern "C" bindings, à la rust-bindgen)
```

C bindings **do** need a shim (`mylib_sys.rs`), because the Jux call must reach a C symbol through Rust's `extern "C"` (§C.9.2, `@extern` row). `bindgen` emits both the stub and the shim; the build links the C library per `jux.toml` (§G.11, §16.5).

### G.7.2. Struct-Pointer Methods Fold to Classes

A family of C functions whose first parameter is `StructName*` folds into a Jux `class` with methods, following the standard `*-sys` → safe-wrapper pattern (§8.1):

```c
// SDL.h
SDL_Window* SDL_CreateWindow(const char* title, int w, int h);
void        SDL_DestroyWindow(SDL_Window* win);
void        SDL_ShowWindow(SDL_Window* win);
```

```jux
// SDL.jux.d
@extern(lib = "SDL2")
public class SDL_Window {
    public SDL_Window(String title, int w, int h);   // SDL_CreateWindow
    public void show();                                // SDL_ShowWindow(this)
    // SDL_DestroyWindow wired to the drop block by the shim
}
```

The leading `SDL_` prefix is stripped (it is now namespaced under the class), `CreateWindow` becomes the constructor, the destructor function is wired to a `drop` block (§6.6) in the generated safe wrapper.

### G.7.3. Free C Functions

A C function not associated with a struct pointer maps to a Jux free function inside a class named for the library (so the namespace survives):

```c
void SDL_Init(uint32_t flags);
```

```jux
@extern(lib = "SDL2")
public class SDL {
    public static void init(uint flags);     // SDL_Init
}
```

### G.7.4. C Type Mapping

| C type            | Jux type     | Notes                                  |
|-------------------|--------------|----------------------------------------|
| `int`             | `i32`        | C `int` is 32-bit on supported targets |
| `unsigned int`    | `u32`        |                                        |
| `long` / `size_t` | `int` / `uint` | platform-sized                       |
| `char*` (string)  | `String`     | assumed NUL-terminated UTF-8; `CString` at the boundary (§5.5) |
| `const char*`     | `String`     |                                        |
| `void*`           | `T*` / opaque | raw pointer, `unsafe` only (§5.5)     |
| `struct Foo*`     | `class Foo`  | folded per §G.7.2                      |
| `struct Foo` (by value) | `struct Foo` | `@layout(c)` (§C.9.2)              |
| function pointer  | `(A) -> R`   | function type (§7.9)                   |
| `enum`            | `enum` with `@layout(c, repr=...)` | §7.7.1 Pattern B     |

### G.7.5. Ownership at the C Boundary

A C pointer the Jux side **owns** (must free) gets its destructor wired to the wrapper's `drop` block (§6.6). A C pointer the Jux side merely **borrows** (the C library owns it) is exposed as a non-owning handle and gets no `drop`. `bindgen` cannot infer ownership from a header alone; the default is **borrow** (safer — no double free), and ownership is opted in per-symbol via the config (§G.11). This is the one place a binding requires human input; the config file documents it explicitly.

## §G.8 — C++ Header Bindings

### G.8.1. Pipeline

```
MyEngine.hpp
    ↓  autocxx (name mangling, vtables, overloads)
    ↓  juxc bindgen --cpp
myengine.jux.d
    +
myengine_sys.rs    (autocxx-generated Rust bindings)
```

C++ adds name mangling, overloading, and member functions over C. `autocxx` handles the mangling and the `extern "C++"` surface; `bindgen` maps the resulting API to Jux stubs.

### G.8.2. C++ Classes → Jux Classes

A C++ `class` with public methods maps directly to a Jux `class`; the C++ constructor maps to a Jux constructor, methods to instance methods, the destructor to a `drop` block:

```cpp
class Renderer {
public:
    Renderer(int width, int height);
    void drawFrame();
    void setTitle(const std::string& title);
};
```

```jux
@extern(lib = "myengine")
public class Renderer {
    public Renderer(int width, int height);
    public void drawFrame();
    public void setTitle(String title);
    // ~Renderer() wired to drop by the shim
}
```

### G.8.3. Overloads

C++ overloads (same name, different parameter lists) map to Jux overloads (§7.2), resolved by the rules in `JUX-TYPE-SYSTEM-ADDENDUM.md` §T.3. C++ default arguments map to Jux default arguments (§7.2).

### G.8.4. Templates

C++ templates are **not** auto-instantiated by `bindgen`. A template is bound only for the explicit instantiations named in the config (mirroring `autocxx`'s requirement). Each named instantiation becomes a distinct Jux generic-applied stub.

## §G.9 — Resolver Ingestion Mode

This is the single compiler change the addendum requires; everything else reuses existing machinery.

### G.9.1. The `.jux.d` Flag

`juxc-resolve` (phase 4, `JUX-COMPILER-PIPELINE-ADDENDUM.md` §C.2.4) gains a per-file flag: a file with the `.jux.d` extension is **external/opaque**. Its declarations are loaded into the symbol table, but they are marked `external`, which means:

- They contribute names, types, and signatures to resolution and to `juxc-tycheck` (phases 6–9).
- They are **excluded from reachability** (phase 9.5, §C.4.5) as roots — a stub is reachable only if user code references it, exactly like any other declaration.
- They **never enter MIR construction** (phase 10): an `external` declaration has no body to lower.
- They are therefore invisible to borrow inference (phase 11), drop/refcount lowering (phases 12–14), and codegen (phase 19+).
- Their **own diagnostics are suppressed** — see §G.9.1.1.

#### G.9.1.1. Stub Diagnostics Are Suppressed (Trust, Don't Validate)

A stub is a *trusted, signature-only view* of an API the foreign toolchain has **already compiled**. Validating it against Jux's rules is both unnecessary (the real crate is the source of truth) and counter-productive at scale: a faithful machine surface of a real Rust crate inevitably references types the generated surface didn't pull in, spells an optional value-type as a Jux-illegal `uint?`, leaves a `const` initializer the parser can't resolve, and so on. None of these are user-actionable, so `juxc-driver` **drops every lex/parse/resolve/tycheck diagnostic attributed to a `.jux.d` source** before reporting (and before deciding whether codegen may proceed). A stub that is 95% well-formed still contributes its 95% to completion and resolution, rather than burying the user in errors about `std`.

This is the mechanism that makes "autocomplete over *any* crate" robust: the bar a stub must clear is **parse far enough to enter the symbol table**, not **type-check clean**. (Diagnostics in ordinary `.jux` sources — including the hand-written `jux.std/` tree — are never suppressed.)

### G.9.2. Lowering Targets

When user code calls an `external` declaration, codegen (§C.9) does not emit a body for it — it emits a reference to the **real** symbol:

- For a Rust-crate stub: the real crate item (no shim — §G.6.2).
- For a C/C++ stub: the symbol exported by the generated `*_sys` shim (§G.7.1, §G.8.1).

The mapping from Jux stub name to real symbol is recorded in a sidecar emitted alongside the `.jux.d` (the shim's symbol table for C/C++, or the rustdoc item path for Rust). Codegen consults it.

### G.9.3. No Borrow-Check of Foreign Bodies

Because stubs have no bodies, the borrow checker never analyzes foreign code — it trusts the signature. The `&` / `&mut` disposition that `bindgen` dropped (§G.3.4) is re-attached at the **call site** in the user's code: a call to a `&mut self` method requires exclusive access to the receiver (§6.3), inferred normally. The foreign implementation is assumed to honor its Rust-checked contract. This is sound because the foreign code is itself borrow-checked by `rustc` at the Phase-1 boundary (§C.9.1).

### G.9.4. Where Stubs Are Found

The resolver discovers `.jux.d` files through the `requires` entries of `module.jux` (§4.3) and the dependency table of `jux.toml` (§G.11). A `requires rust.serde_json` directive points the resolver at `serde_json.jux.d`; a `requires c.sqlite3` points it at `sqlite3.jux.d`. Stubs not named by a `requires`/dependency are not loaded (they cost nothing if unused — §2.5).

## §G.10 — LSP Interaction

### G.10.1. Free Editor Support

Per `JUX-LSP-SERVER-ADDENDUM.md` §L, `juxc-lsp` reuses `juxc-resolve` and `juxc-tycheck` directly and contains no front end of its own. Because stubs are resolved and type-checked by those same crates (§G.9), the LSP answers foreign-symbol queries **with no binding-specific code**:

| LSP feature                | How it works on a foreign symbol                          |
|----------------------------|-----------------------------------------------------------|
| Completion (`textDocument/completion`) | `.` after a `HashMap` value lists its stub methods — the tycheck context already holds them |
| Hover (`textDocument/hover`)           | the recorded signature type is formatted as Jux syntax     |
| Goto-definition (`textDocument/definition`) | jumps to the declaration in the `.jux.d` file (§L.1 goal #3) |
| Document symbols           | a stub file's declarations populate the outline like any file |
| Diagnostics                | a misuse of a foreign signature is an ordinary `juxc-tycheck` error |

### G.10.2. Goto-Definition Lands in the Stub

Go-to-definition on `HashMap` opens the `.jux.d` file at the `HashMap` declaration — the Jux-syntax view, not the Rust source. This is intentional: the programmer reads the API in the syntax they write in. A future enhancement (§G.13) may offer a secondary "go to Rust source" action.

### G.10.3. Stubs Are Indexed Like Sources

The LSP's in-memory document store (`JUX-LSP-SERVER-ADDENDUM.md` §L.2, `doc.rs`) holds `.jux.d` files alongside `.jux` files. They are re-analyzed on change like any document, though in practice they change only when `juxc bindgen` regenerates them (on a dependency bump), not during editing.

## §G.11 — `jux.toml` and Stub Layout

### G.11.1. Declaring Bound Dependencies

Foreign dependencies are declared in `jux.toml` (§12.2) and trigger stub generation:

```toml
[dependencies]
"rust.serde_json" = "1"
"rust.tokio"      = { version = "1", features = ["full"] }
"c.sqlite3"       = { lib = "sqlite3", header = "sqlite3.h" }
"cpp.myengine"    = { lib = "myengine", header = "MyEngine.hpp", instantiate = ["Vec<int>"] }
```

### G.11.2. Generated Stub Location

Generated stubs live in a build-managed directory, not in source control by default:

```
project/
├── src/
│   └── main.jux
├── .jux-stubs/                 ← generated; git-ignored by default
│   ├── rust/
│   │   ├── serde_json.jux.d
│   │   └── tokio.jux.d
│   ├── c/
│   │   └── sqlite3.jux.d
│   └── cpp/
│       └── myengine.jux.d
└── jux.toml
```

`juxc build` regenerates stale stubs (keyed by dependency version + header mtime) before resolution. `juxc bindgen` can also be run manually to emit stubs into a chosen path for inspection or vendoring.

### G.11.3. Per-Symbol Overrides

The two places `bindgen` cannot infer a decision — C-pointer ownership (§G.7.5) and constructor-vs-static for `Self`-returning functions (§G.5.2) — are configured per-symbol in a `[bindgen.<dep>]` table:

```toml
[bindgen."c.sqlite3"]
# sqlite3_open returns a handle the caller must close → Jux owns it
"sqlite3_open".ownership = "owned"
"sqlite3_close".drop_for = "RawHandle"

[bindgen."rust.std"]
"Duration::from_secs".shape = "static"     # keep distinct from_* names
```

## §G.12 — Diagnostics

`bindgen`-related diagnostics use the resolution and backend ranges (`JUX-DIAGNOSTICS-ADDENDUM.md` §D.3). New codes:

| Code  | Severity | Condition                                                          |
|-------|----------|--------------------------------------------------------------------|
| `E0905` | error  | Cannot resolve a bound dependency (existing; reused for stub-source fetch failure) |
| `E0306` | error  | Stub declaration name collides with a user declaration in the same package |
| `W0305` | warning | Two foreign symbols transformed to the same Jux name; second renamed with suffix (§G.4.1) |
| `W0306` | warning | C-pointer ownership defaulted to `borrow`; specify `ownership` in `jux.toml` to silence (§G.7.5) |
| `E0907` | error  | A `.jux.d` declaration has a non-elided body (stubs must be signature-only) |
| `E0908` | error  | A C++ template was referenced without an explicit `instantiate` entry (§G.8.4) |
| `W0307` | warning | A foreign item was skipped (Rust macro, un-mappable type); listed for visibility (§8.2) |

`E0907` is the one stub-specific structural check: a `.jux.d` file whose function has a `{ ... }` body rather than `;` is rejected — bodies belong in the foreign implementation, not the stub.

## §G.13 — Open Questions

- **First-class `import rust.X` vs pre-generated stubs.** §8.2 Layer 3 envisions the compiler reading Rust signatures directly, with no pre-generated `.jux.d`. This addendum specifies the pre-generated path (Layers 2). The two should converge: the in-compiler reader (Layer 3) should produce the *same* in-memory `external` declarations the `.jux.d` resolver produces, so `juxc-lsp` behavior is identical either way. The shared interface between them is unspecified here.
- **Goto-Rust-source.** §G.10.2 lands goto-def in the Jux stub. A secondary action that opens the underlying Rust/C/C++ definition would help advanced users but requires the LSP to hold a stub → source-location map. Deferred.
- **Stub staleness across compiler versions.** A `.jux.d` generated by one `juxc` version may not match another's mapping rules if this addendum's table changes. Stubs should carry a `// bindgen-version: N` header and be regenerated on mismatch. The regeneration trigger is unspecified.
- **Trait-object identity through refcount.** §G.3.1 maps `Box<dyn Trait>` to a Jux interface and `Arc<T>` to `T`. The interaction — a `Box<dyn Trait>` that is itself `Arc`-shared — needs a worked lowering example against §C.9.3 (class lowering) before it is normative.
- **Partial-crate binding.** Binding a whole crate when the user reaches a handful of items wastes `rustdoc` and stub-generation time. Reachability (§C.4.5) strips the unused stubs from the build, but the *generation* cost is paid up front. A lazy, on-demand stub generator (generate only the items resolution actually requests) is plausible but unspecified.
