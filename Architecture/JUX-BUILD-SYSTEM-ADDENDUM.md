# Jux Spec Addendum — Build System and Module Layout

**Status:** Proposed insertion. Specifies the project manifest (`jux.toml`), the module declaration file (`module.jux`), the filesystem-to-package mapping, module discovery and resolution, the dependency-resolution algorithm, the lockfile format, workspace layout, and feature flags. This is the smallest blocking item before a Phase 1 implementer can read any project.

## Design Philosophy (Non-Normative)

Cargo is the most-loved tooling story in modern languages. Jux copies its design wholesale and changes only the file names and the dependency syntax. The reasons:

- Cargo's manifest format is a known good. Yet-another-format invites bikeshedding.
- Cargo's dependency-resolution algorithm (semver + lockfile) handles real-world ecosystems at scale.
- Cargo's workspace model is the smallest viable mono-repo design.
- Cargo's feature-flag system has known bugs (feature unification across the workspace) but the alternatives are worse. Jux inherits the model and makes the same trade-off.

What Jux changes from Cargo:

- File names: `jux.toml` instead of `Cargo.toml`, `jux.lock` instead of `Cargo.lock`.
- Default registry name: `jux-registry.io` instead of `crates.io`.
- Module-vs-package distinction (per JUX-LANG-V1 §4.3): a Jux package may contain multiple modules; Cargo's "crate" maps roughly to "package" here.

## §B.1 — File and Directory Layout

A standard Jux project looks like:

```
my-project/
├── jux.toml                           # project manifest
├── jux.lock                           # generated by `jux build`; commit to VCS
├── module.jux                         # module declaration (optional for single-module projects)
├── src/
│   ├── main.jux                       # entry-point file, contains top-level statements
│   └── com/
│       └── example/
│           └── myapp/
│               ├── Foo.jux            # package com.example.myapp
│               └── Bar.jux
├── test/
│   └── com/
│       └── example/
│           └── myapp/
│               └── FooTest.jux        # @Test functions; mirrors src/ structure
├── examples/                          # optional standalone examples
│   └── basic.jux
├── docs/                              # optional generated/written docs
└── target/                            # build outputs (do not commit)
    ├── debug/
    └── release/
```

### B.1.1. Source-to-Package Mapping

Per JUX-LANG-V1 §3.1 and §4.1, the directory under `src/` mirrors the package path. A file `src/com/example/myapp/Foo.jux` declares `package com.example.myapp;` at its top.

Rules:

- The `src/` directory itself has no package — files directly inside `src/` are package-less. Convention: only `main.jux` lives directly in `src/`.
- Subdirectories of `src/` correspond to package path segments separated by `.`.
- A file's top-level `package` declaration **must** match its directory location, or the build fails (`E0301`).

**Enforcement (juxc, pre-codegen).** A file's package identity is *derived from its path under the source root*; `juxc` validates the written `package` line against that derived value before lowering, so a stale layout never reaches `rustc`. Concretely, for a file under a `src/` root:

| File location | `package` declaration | Result |
|---|---|---|
| directly in `src/` | absent | OK — package-less (root) file |
| directly in `src/` | present | `E0301` — a root file must be package-less |
| `src/a/b/` | absent | `E0301` — missing required `package a.b;` |
| `src/a/b/` | `package a.b;` | OK |
| `src/a/b/` | `package a.x;` (mismatch) | `E0301` — *expected `a.b`* |

The diagnostic points at the offending `package` line (or the file head for a missing declaration) and names the expected package. This is what turns the classic failure — a file in `…/other/` declaring `package xss.it;` while a consumer writes `import xss.it.other.Other;` — from a cryptic emitted-Rust `E0432` into a precise Jux error. Files **not** under a `src/` root (a loose `juxc foo.jux`, the auto-loaded `jux.std`, `.jux.d` stubs) are unconstrained: their package comes from the declaration alone.

### B.1.2. Test Layout

Tests live in a parallel `test/` directory mirroring `src/`. A file `test/com/example/myapp/FooTest.jux` is associated with `src/com/example/myapp/Foo.jux` for tooling purposes (run-tests-for-this-source navigation, coverage report grouping). The test file itself declares its own package, conventionally `com.example.myapp.test` (the production package's name with `.test` appended).

`juxc test` discovers `@Test`-annotated functions across `test/` and runs them.

### B.1.3. Examples Directory

Files in `examples/` each compile as separate runnable programs. Each must contain a `main` function or top-level statements. They share dependencies with the main project but are not built unless explicitly requested via `jux build --examples` or `jux run --example NAME`.

## §B.2 — `jux.toml`: Project Manifest

`jux.toml` is the project's declarative manifest. It uses TOML syntax (per `https://toml.io/en/v1.0.0`).

### B.2.1. Minimum-Viable Manifest

```toml
[package]
name = "com.example.myapp"
version = "0.1.0"
edition = "2026"
```

A manifest with only this section produces a buildable single-module project, no dependencies, default profile.

### B.2.2. Full Schema

```toml
[package]
name = "com.example.myapp"             # REQUIRED. Reverse-DNS package name; lowercase.
version = "0.1.0"                       # REQUIRED. SemVer (https://semver.org).
edition = "2026"                        # REQUIRED in v0.1+. The language edition.

description = "A Jux application."      # Optional. One-line description.
authors = ["A. Programmer <a@x.com>"]   # Optional. List of strings.
license = "MIT"                         # Optional. SPDX license identifier.
license-file = "LICENSE"                # Optional. Path; alternative to `license`.
homepage = "https://example.com/myapp"  # Optional.
repository = "https://github.com/example/myapp"
documentation = "https://docs.example.com/myapp"
readme = "README.md"                    # Optional. Path.
keywords = ["cli", "tool"]              # Optional. Up to 5 strings.
categories = ["command-line-utilities"] # Optional. Registry categories.

# Binary resource metadata. `[package]` `version`/`authors`/`description`/
# `license`/`homepage` flow into every emitted target's resource block.
# Executables additionally embed an icon and a Windows version-info resource
# (CompanyName, FileVersion, ProductVersion, FileDescription, LegalCopyright)
# via the compiler-generated build script. Per-target overrides live on the
# `[lib]` / `[[bin]]` block.
icon = "assets/app.ico"                 # Optional. Windows executable icon (.ico).
company = "XTREME SOFTWARE SOLUTIONS"   # Optional. CompanyName resource (defaults to authors).
copyright = "© 2026 XSS"                # Optional. LegalCopyright resource.

# Code signing (optional). Reserved so an executable target can be signed as a
# post-build step when required — e.g. Authenticode on Windows (signtool /
# osslsigncode) or codesign on macOS. The toolchain runs the configured signer
# on the produced binary after `cargo build`; absence of `[package.sign]` (or
# `[[bin]].sign`) means no signing. Per-target overrides on `[lib]` / `[[bin]]`.
[package.sign]
# enabled    = false                    # Default. Set true to sign the produced exe.
# tool       = "signtool"               # or "osslsigncode" / "codesign" / a custom command.
# certificate = "path/to/cert.pfx"      # or a store identity / thumbprint.
# timestamp-url = "http://timestamp..." # Optional RFC-3161 timestamp authority.
# Credentials (e.g. the .pfx password) come from the environment, never the
# manifest. This block is a reserved seam; the signing step is a later addendum.

[lib]
# A library target. Optional; absent for executable-only projects.
path = "src/lib.jux"                    # Optional. Default: src/lib.jux if exists.
name = "myapp_core"                     # Optional. Default: package name's last segment.
crate-type = ["dylib", "staticlib"]     # Optional. What artifacts to produce.

[[bin]]
# An executable target. Multiple [[bin]] blocks are allowed.
name = "myapp"                          # REQUIRED for explicit binaries.
path = "src/main.jux"                   # Optional. Default: src/main.jux if exists.
main = "xss.it.Main"                    # Optional. Names the entry file by its
                                        # DOTTED source path: "xss.it.Main" ⇒
                                        # src/xss/it/Main.jux (package xss.it).
                                        # Takes precedence over `path`. The build
                                        # selects the `main` declared in that
                                        # package, and editor tooling resolves
                                        # "go to entry point" from the same key.

[dependencies]
# Jux-package dependencies, identified by reverse-DNS name.
"com.mylang.json" = "1.0"
"com.mylang.collections" = { version = ">=2.0, <3.0", features = ["fast"] }
"local-utils" = { path = "../utils" }
"experimental" = { git = "https://git.example.com/exp.git", branch = "main" }

# Foreign-language dependencies use a typed prefix.
"rust.serde_json" = "1.0"
"c.sqlite3" = { lib = "sqlite3", header = "sqlite3.h" }
"c.zlib" = { lib = "z", system = true }                  # use system-installed library

[dev-dependencies]
# Available only to test/ and examples/, not to src/.
"com.example.testkit" = "0.4"

[build-dependencies]
# Available only to build scripts (build.jux). Rare.
# (build.jux is a future feature; specified in a later addendum.)

[features]
# Optional features. Each is a list of dependency names or other features.
default = ["json"]
json = ["dep:com.mylang.json"]                            # enables a dependency
yaml = ["dep:com.mylang.yaml"]
binary_protocol = []                                      # no extra dependencies
fast_math = ["dep:c.libm-fast"]
async = ["dep:com.example.runtime"]
all = ["json", "yaml", "binary_protocol", "async"]        # combines other features

[build]
target = "native"                                         # or a target triple
profile = "full"                                          # full | embedded | core (per V1 §2.4)
optimization = "release"                                  # debug | release | size | none
lto = "thin"                                              # off | thin | fat
strip = true                                              # remove debug info from release
incremental = true                                        # cache intermediates
linker-script = "stm32f407.ld"                            # embedded targets

[profile.dev]                                             # debug-build overrides
opt-level = 0
debug = "full"

[profile.release]                                         # release-build overrides
opt-level = 3
lto = "thin"
strip = true

[profile.embedded]                                        # custom profile
extends = "release"
opt-level = "s"                                           # optimize for size
strip = true
panic = "abort"

[workspace]
# Present only in workspace-root manifests. Lists member packages.
members = ["pkg-a", "pkg-b", "tools/*"]
exclude = ["pkg-experimental"]

[publish]
registry = "default"                                      # default | <registry-name>
allow-dirty = false                                       # prevent publish with VCS changes

[[publish.registry]]                                      # custom registry definitions
name = "internal"
url = "https://registry.internal.example.com"
```

### B.2.3. Field Semantics

**`package.name`**. Reverse-DNS, lowercase, dot-separated. Must match the regex `^[a-z][a-z0-9]*(\.[a-z][a-z0-9_]*)+$`. The package name is what consumers reference in their `[dependencies]` section. It is also the default top-level package path for files under `src/`.

**`package.version`**. SemVer 2.0. Pre-release identifiers (`0.1.0-alpha.1`) and build metadata (`0.1.0+build.123`) are permitted.

**`package.edition`**. Currently `"2026"`. Editions are stable forever once published; a project written for `2026` continues to compile under future compiler versions even if the language adds breaking changes via a new edition.

**`[lib]`**. If present, the package produces a library artifact. `crate-type` (a Cargo-ism Jux retains for familiarity) lists the binary formats produced: `"lib"` (Jux-internal), `"dylib"` (`.so`/`.dylib`/`.dll`), `"staticlib"` (`.a`/`.lib`), `"cdylib"` (C-callable shared library, includes the `.h` header).

**`[[bin]]`**. Multiple binaries per package are permitted. Each must have a unique `name`.

**`[dependencies]`**. Dependency names use the reverse-DNS scheme for Jux packages, or a typed prefix (`rust.`, `c.`, `cpp.`) for foreign-language dependencies. The value is either a SemVer version string or a table with options.

Dependency-table options:

- `version = "..."`: SemVer requirement (defaults to compatible-with-major if omitted on a registry dep).
- `path = "..."`: Local filesystem path. Overrides version requirement.
- `git = "...", branch = "..." | rev = "..." | tag = "..."`: Git source. Pinned by branch (moves), rev (immutable), or tag. **Shorthand:** a bare string value that looks like a repository URL (`http://`, `https://`, `git@`, `ssh://` prefix) is the git form tracking the remote's default branch — `"com.x.lib" = "https://github.com/u/repo"` ≡ `"com.x.lib" = { git = "https://github.com/u/repo" }`.
- `registry = "..."`: Non-default registry name (registered in `[[publish.registry]]`).
- `features = ["a", "b"]`: Enable optional features.
- `default-features = false`: Disable the dep's default features.
- `optional = true`: The dep is included only when a feature names it.
- For `c.*` deps: `lib = "name"` (link library name), `header = "..."` (include path), `system = true` (search system paths only, don't bundle).

### B.2.4. Edition Field

The `edition` field selects the language edition. v0.1 supports only `"2026"`. Later editions may rename keywords, change syntax, or alter default behavior; older editions continue to work because the compiler dispatches on edition at parse time.

**Editions are not breakage.** A `2026`-edition crate can depend on a future `2029`-edition crate transparently. Each compiles under its declared edition.

## §B.3 — `module.jux`: Module Declaration

A `module.jux` file at the source-tree root declares a multi-module package. JUX-LANG-V1 §4.3 introduces the syntax; this section completes it.

### B.3.1. Syntax

```jux
module com.mylang.json {
    version "1.0.0";

    exports com.mylang.json;
    exports com.mylang.json.parser;
    // com.mylang.json.internal is NOT exported.

    requires com.mylang.collections version ">=2.0, <3.0";
    requires com.mylang.io;

    requires rust.serde_json;
    requires c.sqlite3 lib "sqlite3";

    feature "fast" enables fast-path;
    feature "logging" requires com.mylang.log;
}
```

### B.3.2. Module-vs-Package Relationship

A package (a `jux.toml`) contains:

- **Zero or one `module.jux`.**
- **One or more source modules.** A "module" in Jux is any package path that contains source files plus all of its sub-paths.

If `module.jux` is absent, the package implicitly:

- Exports every `public` symbol.
- Has no `requires` other than what `[dependencies]` in `jux.toml` lists.
- Has no internal-only sub-modules.

Most simple packages omit `module.jux` and live entirely off `jux.toml`. A package with internal-only sub-paths (per the `internal` visibility level in JUX-LANG-V1 §4.4) needs `module.jux` to declare which sub-paths are exported.

### B.3.3. Module Block Statements

Each statement inside the module block:

- `version "X.Y.Z"`: optional; if absent, taken from `jux.toml`. Useful when `jux.toml` carries a different version (e.g., during pre-release).
- `exports <package-path>`: marks a sub-package as part of the public API. Sub-packages not listed are internal.
- `requires <name> [version "..."] [lib "..."]`: declares a dependency. Must be consistent with `jux.toml`'s `[dependencies]`. The `module.jux` declaration is the source of truth for *which* sub-package depends on which dependency; `jux.toml` is the source of truth for the version.
- `feature "<name>" [enables ...] [requires <name>]`: declares an optional feature, mirrored in `jux.toml`'s `[features]`.

### B.3.4. Empty Modules

A module with no source files (only the `module.jux` declaration) is permitted but warned (`W0210`, "module declares no exported symbols"). Useful for declaring a top-level facade that re-exports from sub-modules.

## §B.4 — Module Discovery

Given a top-level `import com.example.foo.Bar;`, `juxc` resolves the import as follows.

### B.4.1. Resolution Order

For each imported name, search in this order:

1. **Local source.** Files under `src/` of the current package whose `package` declaration matches.
2. **Local dev-dependencies** (test/example contexts only). `dev-dependencies` from `jux.toml`.
3. **Direct dependencies.** `dependencies` from `jux.toml`, in declaration order. The first matching dep wins.
4. **Standard library.** Built-in tier-0 (`core.*`) and tier-1 (`std.*`) packages bundled with the compiler.

If no match is found, emit `E0301` ("name not found"). If multiple matches exist at the same resolution level (e.g., two dependencies both expose the same sub-path), emit `E0303` listing the candidates.

### B.4.2. Wildcard Imports

`import com.example.foo.*;` brings in every public top-level declaration of the named package — but **not** its sub-packages. To import sub-packages, name them explicitly.

### B.4.3. Renamed Imports

`import com.example.foo.Bar as MyBar;` introduces `MyBar` into the current scope as an alias. `Bar` is not introduced. Renames apply only to the importing file.

### B.4.4. Grouped Imports

`import com.example.foo.{Bar, Baz, Quux};` is shorthand for three separate imports. Renames inside groups: `import com.example.foo.{Bar as B, Baz};`.

### B.4.5. Conditional Imports

`@cfg(os = "linux") import std.platform.linux.*;`. Per JUX-LANG-V1 §11.4. The conditional is evaluated at phase 5 (per `JUX-COMPILER-PIPELINE-ADDENDUM.md` §C.2.5), after which non-matching imports vanish.

### B.4.6. Cyclic Imports

A cycle in the dependency graph between source modules is rejected (`E0302`) unless one of the modules is `@async-init` (per JUX-LANG-V1 §10.1.7). Cyclic imports across packages (rather than across source modules within a package) are always rejected.

## §B.5 — Dependency Resolution

`jux build` runs the resolver to produce a concrete set of versioned dependencies and writes the result to `jux.lock`.

### B.5.1. Algorithm

1. **Collect requirements.** Walk the dependency tree, gathering every `[dependencies]` and `[dev-dependencies]` entry. Each entry is a `(package, requirement, source)` tuple where `source` is registry/git/path.
2. **Group by package.** Multiple requirements for the same package are combined.
3. **Solve.** For each package, find the highest version that satisfies every requirement. If no such version exists, error with the conflicting requirements listed.
4. **Determine features.** For each resolved package, compute the union of features requested by all of its requirers (Cargo's "feature unification" rule). All consumers of a package see the same features; this can occasionally cause unexpected feature enablement, but the alternative (per-consumer features) creates monomorphization explosion.
5. **Emit lockfile.** Serialize the resolution to `jux.lock`.

### B.5.2. Version Requirement Syntax

| Form              | Meaning                                                  |
|-------------------|----------------------------------------------------------|
| `"1.2.3"`         | Compatible with `1.2.3` per SemVer (`>=1.2.3, <2.0.0`)   |
| `"^1.2.3"`        | Same as above (explicit caret)                          |
| `"~1.2.3"`        | `>=1.2.3, <1.3.0`                                       |
| `"=1.2.3"`        | Exactly `1.2.3`                                         |
| `">=1.2.3"`       | Any version `>= 1.2.3`                                  |
| `">=1.2.3, <2.0"` | Range with comma-separated constraints                  |
| `"*"`             | Any version. Discouraged; `jux publish` rejects.        |
| `"1"`, `"1.2"`    | Compatible with `1.0.0` or `1.2.0` (Cargo shortcuts)    |

### B.5.3. Pre-release Versions

Pre-release versions (`0.1.0-alpha.1`) require explicit opt-in: `"0.1.0-alpha.1"` or `">=0.1.0-alpha.1, <0.2.0"`. A `"^0.1.0"` requirement does **not** match `0.1.0-alpha.1` even though the latter sorts higher; pre-releases are excluded by default. This matches Cargo and SemVer-preferred-by-most-tooling behavior.

### B.5.4. Conflicts

When two requirers ask for incompatible versions of the same package:

```
Error E0905: cannot resolve dependency `com.mylang.json`
  Required by `pkg-a` as `^1.5`
  Required by `pkg-b` as `^2.0`
  No version satisfies both ranges.
Hint: align `pkg-a` and `pkg-b` on the same major version, or vendor one of them.
```

The resolver does not attempt to pick a "least-bad" version. Either both requirers agree, or the user must intervene.

### B.5.5. Source Priority

If the same package is referenced from multiple sources (registry, git, path), the **highest-priority source wins**, with priority ordered: path > git > registry. This allows local development overrides.

For each direct dep, the user can also use `[patch]`-equivalent (later addendum) to override transitive deps.

## §B.6 — Lockfile (`jux.lock`)

After resolution, `juxc` writes `jux.lock` recording the exact versions chosen:

```toml
# jux.lock — generated by jux build. Do not edit manually.

version = 1                                              # lockfile format version

[[package]]
name = "com.example.myapp"
version = "0.1.0"
source = "local"
dependencies = [
    "com.mylang.json 1.4.2 (registry+default)",
    "com.mylang.collections 2.7.0 (registry+default)",
]

[[package]]
name = "com.mylang.json"
version = "1.4.2"
source = "registry+default"
checksum = "sha256:abc123..."
dependencies = [
    "com.mylang.collections 2.7.0 (registry+default)",
]

[[package]]
name = "com.mylang.collections"
version = "2.7.0"
source = "registry+default"
checksum = "sha256:def456..."
dependencies = []
```

### B.6.1. Lockfile Rules

- **Library packages do not commit `jux.lock`.** Their consumers re-resolve. Cargo convention.
- **Application packages do commit `jux.lock`.** Reproducible builds.
- **A `jux.lock` from a different compiler version is migrated forward,** preserving as much resolution as possible. Migration is automatic.
- **Lockfile checksums are verified on every build.** A registry that serves a tampered package fails the build.

### B.6.2. Updating the Lockfile

`jux update` re-runs resolution. Without arguments, updates all dependencies to the latest compatible versions. With a name, updates only that package: `jux update com.mylang.json`.

`jux update --aggressive` ignores compatible-version constraints and updates to the latest version of every dep, then re-checks compatibility. Used for major-version upgrades.

## §B.7 — Workspaces

A workspace is a set of related packages built together with a shared lockfile and shared dependencies.

### B.7.1. Workspace Layout

```
my-workspace/
├── jux.toml                           # workspace-root manifest
├── jux.lock                           # shared lockfile
├── pkg-a/
│   ├── jux.toml
│   └── src/
├── pkg-b/
│   ├── jux.toml
│   └── src/
└── tools/
    └── codegen/
        ├── jux.toml
        └── src/
```

The workspace-root `jux.toml`:

```toml
[workspace]
members = ["pkg-a", "pkg-b", "tools/*"]
exclude = ["pkg-experimental"]
default-members = ["pkg-a", "pkg-b"]            # what `jux build` builds by default

[workspace.dependencies]
# Common dependencies usable across the workspace via `workspace = true`.
"com.mylang.json" = "1.0"
"com.mylang.collections" = "2.0"

[workspace.package]
# Common package metadata.
edition = "2026"
authors = ["Example Team"]
license = "MIT"
```

A member package references workspace-level entries:

```toml
[package]
name = "com.example.pkg-a"
version = "0.1.0"
edition.workspace = true
license.workspace = true

[dependencies]
"com.mylang.json".workspace = true
```

### B.7.2. Workspace Resolution

`jux build` at the workspace root builds every default member. `jux build -p pkg-a` builds only `pkg-a`. The resolver runs once, producing a single `jux.lock` consumed by all members. This guarantees that every member sees the same version of any shared dependency — no diamond dependency conflicts within a workspace.

### B.7.3. Why Workspaces Are Standard

A monorepo of related Jux packages — a server + its client + their shared protocol — should live in one workspace. The shared lockfile and unified `jux build` make iteration fast and prevent version skew. This is the design that makes Cargo's monorepo story work; Jux inherits it directly.

## §B.8 — Feature Flags

Features are named build-time switches that enable optional code, dependencies, or APIs. Per JUX-LANG-V1 §11.6.

### B.8.1. Declaration

In `jux.toml`:

```toml
[features]
default = ["json", "logging"]                           # enabled if the consumer says nothing
json = ["dep:com.mylang.json"]                           # enables an optional dep
yaml = ["dep:com.mylang.yaml"]
fast_math = ["dep:c.libm-fast"]
all = ["json", "yaml", "fast_math"]                       # combines others
```

In `module.jux`:

```jux
module com.example.app {
    feature "json" enables json-codec;
    feature "yaml" enables yaml-codec;
    feature "all" requires json, yaml;
}
```

### B.8.2. Enabling Features

Consumers specify features in their dependency entry:

```toml
[dependencies]
"com.example.app" = { version = "1.0", features = ["yaml"], default-features = false }
```

`default-features = false` disables the package's `default` set. The example above gets `yaml` only, no `json` or `logging`.

### B.8.3. Code Conditional on Features

```jux
@cfg(feature = "json")
public class JsonCodec { ... }

@cfg(feature = "yaml")
public class YamlCodec { ... }

@cfg(all(feature = "json", feature = "yaml"))
public class AnyFormatCodec { ... }
```

Per JUX-LANG-V1 §11. `@cfg(feature = "X")` evaluates true iff feature `X` is enabled in the current build.

### B.8.4. Feature Unification

When the same dependency appears with different feature sets in different parts of the tree, the resolver computes the **union** of all requested features and enables them all. This matches Cargo. The implication: enabling a feature in one place can affect every consumer of the same dep.

To avoid surprises, libraries should keep their feature sets small and orthogonal. Features should rarely need to be mutually exclusive; if they do, refactor into separate packages.

### B.8.5. Optional Dependencies

A dependency listed with `optional = true` is included only when a feature names it via `dep:<name>`:

```toml
[dependencies]
"com.mylang.fast-json" = { version = "1.0", optional = true }

[features]
fast = ["dep:com.mylang.fast-json"]
```

Without the feature flag, the dependency is not fetched and its symbols are not available.

## §B.9 — Build Profiles in the Manifest

The build profile (per JUX-LANG-V1 §2.4) is set in `[build]`:

```toml
[build]
profile = "full"            # full | embedded | core
target = "native"           # native | <target-triple>
optimization = "release"
```

Custom-named build profiles inherit and override:

```toml
[profile.dev]
opt-level = 0               # no optimization
debug = "full"               # full debug info
overflow-checks = true       # arithmetic overflow panics
incremental = true

[profile.release]
opt-level = 3
debug = false
strip = true
lto = "thin"
overflow-checks = false      # release: wrap on overflow

[profile.embedded]
extends = "release"
opt-level = "s"              # optimize for size
panic = "abort"              # don't unwind in core/embedded
strip = "all"

[profile.test]
opt-level = 1                # fast tests, but readable stack traces
```

### B.9.1. Profile Fields

| Field              | Values                                   | Meaning                                     |
|--------------------|------------------------------------------|---------------------------------------------|
| `opt-level`        | `0`–`3`, `"s"`, `"z"`                    | Optimization level; `s`/`z` for size         |
| `debug`            | `false`, `"line-tables"`, `"full"`       | Debug-info level                            |
| `strip`            | `false`, `"debuginfo"`, `"all"`          | Strip symbols from output                    |
| `lto`              | `"off"`, `"thin"`, `"fat"`               | Link-time optimization                       |
| `overflow-checks`  | `true`, `false`                          | Trap on integer overflow                     |
| `panic`            | `"unwind"`, `"abort"`                    | Panic strategy                              |
| `incremental`      | `true`, `false`                          | Cache intermediates                          |
| `codegen-units`    | integer or `"max"`                       | Parallelism of codegen                       |
| `extends`          | profile name                             | Inherit from another profile                 |

The default profiles are `dev` (used by `jux build` without `--release`) and `release` (used by `jux build --release`). `test` is used by `jux test`. `bench` is used by `jux bench` (when added).

### B.9.2. Phase-1 Status & Divergences

`[build] optimization` and the `[profile.*]` tables are **implemented**. They lower into the per-module emitted `Cargo.toml`, so each module (workspace member) controls its own profiles and build type independently.

- **`[build] optimization`** selects the default build type when no CLI flag is given: `"release"`/`"size"` build optimized, `"debug"`/`"none"` (and absence) build debug. An explicit `jux build --release` always wins. `"size"` additionally injects `opt-level = "z"` into the emitted `[profile.release]` unless the manifest already pins one.
- **`[build] target`** supplies a default cross-compile triple (overridden by the CLI `--target`); `"native"` is normalized to "no triple".
- **`[profile.*]`** keys are translated to their Cargo spellings on emission: `debug = "line-tables"` → Cargo `"line-tables-only"`; `strip = "all"` → Cargo `"symbols"`; spec `extends` → Cargo `inherits` (and Cargo's `inherits` spelling is also accepted). A custom-named profile (anything but `dev`/`release`/`test`/`bench`) always emits an `inherits` (defaulting to `"dev"`), since Cargo requires it.
- **Divergences:** custom-named profiles are emitted but not yet *selectable* (no `jux build --profile <name>` flag — only `dev`/`release` are reachable, via `--release`). A non-integer `codegen-units` (e.g. the spec's `"max"`) is dropped on emission because Cargo only accepts an integer there.

## §B.10 — Registry Protocol

Phase 1 supports a single default registry plus arbitrary user-configured registries. The registry protocol matches Cargo's HTTP+TOML protocol with renamed paths.

### B.10.1. Registry Index

Each registry exposes an HTTP endpoint that lists packages and their versions:

```
GET https://<registry>/index/<package-name-prefix>/<package-name>
  → text response, one JSON line per published version:
    {"name":"...","vers":"...","deps":[...],"features":{...},"checksum":"..."}
```

The package-name-prefix is a sharding scheme based on the package name's hash, identical to crates.io's. Prevents directories with millions of files.

### B.10.2. Crate Download

```
GET https://<registry>/crates/<package-name>/<version>.jux-pkg
  → binary response, a `.jux-pkg` archive (gzipped tar of the source tree)
```

### B.10.3. Authentication

Public registry: read-only access is anonymous. Publishing requires authentication via API token, configured in `~/.jux/credentials.toml`:

```toml
[registries.default]
token = "ABC123..."
```

### B.10.4. Publishing

`jux publish` packages the current directory and uploads:

1. Verify `jux.toml` is well-formed.
2. Verify `jux.lock` is up to date.
3. Run `jux check` to ensure the package compiles.
4. Run `jux test` to ensure tests pass.
5. Verify VCS is clean unless `--allow-dirty` was passed.
6. Build a `.jux-pkg` archive of the source tree.
7. Compute checksums.
8. Upload to the registry.

The registry verifies the package signature, scans for known-bad patterns, and stores the package immutably. Once published, a (package, version) pair cannot be deleted or changed; it can only be **yanked** (marked as "do not use for new resolutions"). Yanked versions remain available for existing lockfiles.

### B.10.5. The `jux` Binary's Subcommands

```
jux new <name>                          # create a new project skeleton
jux init                                # add jux.toml to existing source tree
jux add <pkg>                            # add a dependency
jux remove <pkg>                         # remove a dependency
jux build [--release] [--target <triple>] [-p <pkg>]
jux check
jux test [-p <pkg>] [-- <test-filter>]
jux bench
jux run [--bin <name>] [-- <args...>]
jux update [<pkg>]
jux clean
jux publish [--registry <name>]
jux yank <pkg>@<version>
jux fmt
jux doc [--open]
jux tree                                 # dependency tree
jux audit                                 # check for known vulnerabilities (stub in v0.1)
```

These dispatch to `juxc` for actual compilation. `jux` is the project-level orchestration layer, `juxc` is the compiler.

## §B.11 — `juxc` vs `jux`

Per JUX-LANG-V1 §12.1–§12.2, there are two binaries:

- **`juxc`** — the compiler. Operates on individual files or modules. Doesn't know about manifests or dependencies.
- **`jux`** — the project tool. Reads `jux.toml`, resolves dependencies, dispatches `juxc` invocations.

A user typically only runs `jux`. `juxc` is invoked by `jux` and by tooling (LSP, CI scripts) that needs finer control. The two binaries ship together in the standard distribution.

### B.11.1. Why Two Binaries

This separation:

- Lets the language server (`juxc lsp`) skip dependency resolution entirely, providing fast feedback on a single file.
- Lets the project tool be replaced (advanced users, alternative build systems) without replacing the compiler.
- Mirrors `cargo`/`rustc`, which is well understood.

A future Bazel/Buck integration would call `juxc` directly. A future "single-file script" mode would also bypass `jux`.

## §B.12 — FFI Wrapper Conventions

Each wrapped C library uses a **two-file pattern** under `src/ffi/<library>/`:

```
src/ffi/<library>/
├── raw.generated.jux           # auto-generated, do not edit
└── <WrapperName>.jux           # hand-written safe wrapper
```

Example for SQLite:

```
src/ffi/sqlite/
├── raw.generated.jux
└── Database.jux
```

### B.12.1. The Generated File

`raw.generated.jux` is produced by `jux bindgen`. Its header records the source header path and a content hash for cache validation:

```java
// AUTO-GENERATED — DO NOT EDIT
// Generated from: /usr/include/sqlite3.h
// Hash: a3f5b9c2...
// Run `jux bindgen sqlite3` to regenerate.

@NativeModule
@Native(lib = "sqlite3")
package com.example.app.ffi.sqlite.raw;

@Extern type sqlite3;
@Extern type sqlite3_stmt;

@Extern int sqlite3_open(CString filename, out sqlite3* db);
@Extern int sqlite3_close(sqlite3* db);
@Extern int sqlite3_prepare_v2(sqlite3* db, CString sql, int nByte,
                                out sqlite3_stmt* stmt, out CString tail);
// ... hundreds of declarations
```

### B.12.2. The Safe Wrapper

The hand-written wrapper converts C idioms (error codes, raw pointers, manual cleanup) to Jux idioms (exceptions, handles, RAII via `drop`):

```java
package com.example.app.ffi.sqlite;

import com.example.app.ffi.sqlite.raw.*;

public class SqlError extends Exception {
    public SqlError(String message) { super(message); }
}

public final class Database {
    private sqlite3* handle;

    public Database(String path) throws SqlError {
        var h: sqlite3* = null;
        var rc = sqlite3_open(path.toCString(), out h);
        if (rc != 0) {
            throw new SqlError($"Failed to open: $path (code $rc)");
        }
        this.handle = h;
    }

    public Statement prepare(String sql) throws SqlError {
        var stmt: sqlite3_stmt* = null;
        var tail: CString = null;
        var rc = sqlite3_prepare_v2(handle, sql.toCString(), -1, out stmt, out tail);
        if (rc != 0) throw new SqlError($"Prepare failed: $sql");
        return new Statement(stmt);
    }

    drop {
        sqlite3_close(handle);
    }
}
```

### B.12.3. Git Policy: Commit Generated Files

`raw.generated.jux` is **committed** to the repository. Rationale:

- Builds work without libclang installed on contributor machines.
- CI doesn't need a C toolchain.
- Faster clean builds.
- Diffs of binding changes are visible in pull requests.

The trade-off (occasional noisy diffs when headers change) is worth the reproducibility and contributor-onboarding benefits.

### B.12.4. Cache Location

`jux bindgen` caches its output keyed by header hash:

```
target/
└── .ffi-cache/
    └── <library>/
        ├── <header>.hash       # hash of the header file
        └── raw.generated.jux   # cached output
```

The build tool only re-runs bindgen when the header content changes or the bindgen configuration changes.

### B.12.5. Pre-Wrapped Libraries

For common libraries, Jux ships pre-built wrappers in `std.ffi.*`:

```
std.ffi.sqlite        — SQLite
std.ffi.libc          — POSIX system calls
std.ffi.curl          — HTTP via libcurl
std.ffi.openssl       — TLS / crypto
std.ffi.posix         — Unix system calls
std.ffi.wasm          — WebAssembly runtime (future)
```

Users import these directly without wrapping work:

```java
import std.ffi.sqlite.Database;

var db = new Database("app.db");
```

## §B.13 — Bindgen Regeneration Rules

### B.13.1. Triggers

`jux bindgen` re-runs when any of these change:

- The C header file's content hash.
- The bindgen configuration in `jux.toml`.
- The target platform (different ABI = different bindings).
- The bindgen tool version.
- The user invokes `jux bindgen --force`.

If nothing changed, the cached output is reused.

### B.13.2. Change Classification

When regeneration produces output that differs from the existing `raw.generated.jux`, the tool classifies the change:

**A. No semantic difference.** Silent update (reordering, comments). No notification.

**B. Additions only.**

```
ℹ Regenerated bindings for c.sqlite3
  Added: 3 new functions
    - sqlite3_db_config
    - sqlite3_serialize
    - sqlite3_deserialize
  Hand-written wrapper at src/ffi/sqlite/Database.jux is unaffected.
```

Existing code continues to work; new functions are available if the user chooses to wrap them.

**C. Removals or signature changes.**

```
⚠ Bindings for c.sqlite3 changed in incompatible ways:
  Removed: sqlite3_old_function
  Changed: sqlite3_exec
    parameter count: 5 → 4

This may break src/ffi/sqlite/Database.jux. Please review.
Continue? [y/N]
```

The user must confirm before the new bindings are committed to the cache. This prevents silent breakage of hand-written wrappers.

### B.13.3. Per-Library Configuration

```toml
[ffi.sqlite3]
lib = "sqlite3"
header = "/usr/include/sqlite3.h"
output = "src/ffi/sqlite/raw.generated.jux"

# Filter what gets bound
allowlist_functions = ["sqlite3_.*"]
blocklist_functions = ["sqlite3_deprecated_.*"]
allowlist_types = ["sqlite3.*"]

# Regeneration strategy
regenerate = "auto"              # auto | manual | always
```

#### `regenerate` Modes

| Mode      | Behaviour                                                            |
|-----------|----------------------------------------------------------------------|
| `auto`    | Regenerate when header changes (default)                             |
| `manual`  | Only regenerate when `jux bindgen` is invoked explicitly             |
| `always`  | Regenerate on every build (slow, for testing)                        |

### B.13.4. Missing Function Errors

When a wrapper references a function not in the bindings, the diagnostic points to the resolution:

```
error: cannot find function `sqlite3_new_thing` in module `c.sqlite3.raw`
  --> src/ffi/sqlite/Database.jux:60:9
   |
60 |     sqlite3_new_thing(handle);
   |     ^^^^^^^^^^^^^^^^^ not found
   |
   = help: if this function exists in the C header, run:
            jux bindgen sqlite3 --force
   = help: to manually add it, edit src/ffi/sqlite/raw.generated.jux
```

### B.13.5. Vendored Headers

For libraries where deterministic bindings matter (different system versions produce different bindings), vendor the header:

```
src/ffi/sqlite/
├── headers/
│   └── sqlite3.h              # vendored
├── raw.generated.jux
└── Database.jux
```

```toml
[ffi.sqlite3]
header = "src/ffi/sqlite/headers/sqlite3.h"   # relative, not system
```

Same header content = same bindings, regardless of what's installed on the build machine.

## §B.14 — FFI Linkage Configuration

The `[ffi.<name>]` table controls how a foreign library is linked. Three linkage modes; platform-specific overrides; default linkage per build profile.

### B.14.1. Complete Schema

```toml
[ffi.<name>]
lib = "string"                       # library name (without lib prefix / suffix)
header = "path/to/header.h"          # C header for bindgen
linkage = "dynamic" | "static" | "framework"

# Finding the library file:
lib_path = "path/to/dir/"            # search directory
lib_file = "path/to/libname.a"       # explicit file (alternative to lib_path)

# Additional dependencies:
extra_libs = ["dep1", "dep2"]        # transitive libraries
extra_lib_paths = ["path1/", "path2/"]

# Build from source (advanced):
build_from_source = false
source_path = "path/to/src/"
build_command = "make all"
output_lib = "path/to/result.a"

# Bindgen config (see §B.13):
allowlist_functions = ["pattern.*"]
blocklist_functions = ["internal_.*"]
allowlist_types = ["MyType.*"]
regenerate = "auto"

# Platform-specific overrides:
[ffi.<name>.target.linux]
lib_path = "libs/linux/"

[ffi.<name>.target.macos]
lib_path = "libs/macos/"

[ffi.<name>.target.windows]
lib_path = "libs/windows/"
lib_file = "libs/windows/special.lib"
```

### B.14.2. Linkage Modes

| Mode        | What it does                                                |
|-------------|-------------------------------------------------------------|
| `dynamic`   | Link against `libfoo.so` / `foo.dll` at runtime             |
| `static`    | Bundle `libfoo.a` / `foo.lib` into the binary               |
| `framework` | macOS-specific bundle linking                               |

### B.14.3. Common Cases

**System library (already installed):**

```toml
[ffi.sqlite3]
lib = "sqlite3"
linkage = "dynamic"
# No lib_path — linker searches system paths
```

**Vendored library (shipped with project):**

```toml
[ffi.sqlite3]
lib = "sqlite3"
header = "vendor/sqlite/sqlite3.h"
linkage = "static"
lib_path = "vendor/sqlite/libs/{target.os}-{target.arch}/"
```

Placeholders `{target.os}` and `{target.arch}` are substituted at build time.

**Mixed linkage:**

```toml
[ffi.sqlite3]
lib = "sqlite3"
linkage = "static"               # bundle SQLite

[ffi.openssl]
lib = "ssl"
linkage = "dynamic"              # use system OpenSSL
extra_libs = ["crypto"]
```

### B.14.4. Default Linkage by Profile

| Profile           | Default linkage           |
|-------------------|---------------------------|
| `jux-full`        | dynamic                   |
| `jux-embedded`    | static                    |
| `jux-core`        | static (only option)      |

The user may override per-library, but defaults match the platform's normal usage.

### B.14.5. Build From Source

For libraries easier to build than to vendor as prebuilt binaries:

```toml
[ffi.mylib]
lib = "mylib"
header = "vendor/mylib/include/mylib.h"
linkage = "static"
build_from_source = true
source_path = "vendor/mylib/"
build_command = "make libmylib.a"
output_lib = "vendor/mylib/libmylib.a"
```

The build tool runs `build_command` in `source_path` before linking, then uses `output_lib`.

### B.14.6. Embedded Profile Constraints

On MCU targets, dynamic linking isn't available — everything must be static:

```toml
[build]
profile = "core"
target = "thumbv7em-none-eabihf"

[ffi.stm32_hal]
lib = "stm32_hal"
header = "vendor/stm32/Drivers/STM32F4xx_HAL_Driver/Inc/stm32f4xx_hal.h"
linkage = "static"               # mandatory in core profile
lib_path = "vendor/stm32/lib/"
```

Specifying `linkage = "dynamic"` under the `core` profile is `E0908` (`dynamic linkage unavailable in core profile`).

### B.14.7. Linker Output

The build tool generates appropriate linker flags from the configuration. The user never writes `-L`, `-l`, `-Wl,-Bstatic`, `/LIBPATH:`, or `-framework` directly. Platform differences are handled automatically.

### B.14.8. Documentation Generation

`jux doc` lists FFI dependencies in the generated documentation under a `## Dependencies` section, e.g.:

```
## Dependencies

### Native Libraries
- **sqlite3** (static, vendored at vendor/sqlite/)
- **ssl, crypto** (dynamic, system)

### Jux Packages
- com.mylang.json v1.0
```

Useful for licensing audits and security reviews.

### B.15.2. Multi-Binary Projects

A package may declare more than one binary entry:

```
src/
├── lib.jux                     # shared code
├── main.jux                    # primary binary
└── bin/
    ├── server.jux              # additional binary
    └── migrator.jux            # additional binary
```

```toml
[[bin]]
name = "myapp"
path = "src/main.jux"

[[bin]]
name = "myapp-server"
path = "src/bin/server.jux"

[[bin]]
name = "myapp-migrator"
path = "src/bin/migrator.jux"
```

Each binary may import from `src/lib.jux`. Build all with `jux build`; run a specific one with `jux run --bin myapp-server`. Each `path` must contain exactly one entry point per `JUX-ENTRY-POINTS-ADDENDUM.md` §E.1.3 / §E.2.

### B.15.3. Examples Directory

`examples/` holds standalone runnable programs that demonstrate the library:

```
examples/
├── basic_usage.jux
├── advanced_features.jux
└── README.md
```

Run with `jux run --example basic_usage`. Examples are compiled in CI and act as integration tests.

### B.15.4. Target Directory Layout

Build artifacts live under `target/`, always gitignored, always regenerable:

```
target/
├── debug/                      # debug builds
├── release/                    # release builds
├── doc/                        # generated documentation
├── tests/                      # compiled test binaries
├── .jux-cache/                 # incremental compilation cache
├── .ffi-cache/                 # bindgen cache (see §B.12.4)
└── .rust-build/                # Phase 1: generated Rust + Cargo build
    ├── Cargo.toml              # auto-generated from jux.toml
    ├── src/                    # generated .rs files
    └── target/                 # Cargo's nested target dir
```
